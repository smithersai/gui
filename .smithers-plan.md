# 0195 Beta Happy Path E2E Trace — Implementation Plan

## Scope
- Deliver one checked-in beta happy-path trace that maps user actions to backend assertions for readiness gating.
- Add or update iOS E2E happy-path coverage for: sign-in/restored auth bypass, workspace open, chat send/dispatch, run discovery, approval decision, output/log visibility, terminal behavior (enabled path if 0188 shipped; explicit gated-copy assertion otherwise), and sign-out data wipe.
- Keep harness seeding minimal and bias tests toward production discovery flows over hardcoded `PLUE_E2E_*` IDs, preserving seeded IDs only where still required by known backend constraints.
- Ensure artifacts for failures are preserved (xcresult/log paths) and local one-command execution remains true for iOS and macOS scripts.
- Include macOS happy-path coverage only if remote mode is in active beta scope; otherwise document exclusion in trace/spec.

## Files To Change
- `ios/Tests/SmithersiOSE2ETests/*HappyPath*` (new focused happy-path test file and/or helper extensions).
- `macos/Tests/SmithersMacOSE2ETests/*HappyPath*` (only if macOS remote mode included).
- `ios/scripts/seed-e2e-data.sh` (trim seeded rows/outputs to minimum required; keep idempotent behavior).
- `ios/scripts/run-e2e.sh` (ensure happy-path env wiring, artifact logging clarity, and one-command ergonomics remain intact).
- `macos/scripts/run-e2e.sh` (same artifact/ergonomics checks; align with iOS conventions where useful).
- Trace/spec doc under one of:
  - `docs/` (for tester-facing narrative), or
  - `.smithers/specs/` (for ticket-linked acceptance trace).
- Optional production app touch-ups only for missing accessibility identifiers discovered during E2E authoring.

## Sequencing
1. Baseline + dependency check
- Confirm current 0188 status and choose terminal branch for the trace: enabled-path assertions vs explicit gated-copy assertions.
- Run existing `./ios/scripts/run-e2e.sh` once to capture current behavior and artifact paths.

2. Author checked-in trace document
- Create/update a trace document listing each user step, expected UI state, and corresponding backend assertion/evidence source (API/log/run screen).
- Explicitly document terminal branch condition and macOS inclusion/exclusion decision.

3. Implement iOS happy-path E2E
- Add a single deterministic happy-path spec file focused on user-visible assertions.
- Reuse production discovery paths first (workspace/runs/approvals discovery) and avoid direct seeded-ID coupling unless unavoidable.
- Include sign-out completion and post-sign-out stale-data absence checks.

4. Right-size seed + runner scripts
- Reduce `seed-e2e-data.sh` payload to only data required by the happy path.
- Keep/run script outputs explicit about xcresult/log locations and failure debugging.
- Ensure command remains one-step runnable against local Plue stack.

5. macOS coverage decision + implementation (conditional)
- If remote mode is in scope, add macOS happy-path test parity for critical checkpoints.
- If out of scope, document rationale and keep script/doc references consistent.

6. Final verification + docs sync
- Re-run `./ios/scripts/run-e2e.sh` and `./macos/scripts/run-e2e.sh` (or documented conditional for macOS).
- Verify artifacts are produced/documented and acceptance criteria are mapped in the trace doc.

## Validation Strategy
- Execution validation:
  - `./ios/scripts/run-e2e.sh`
  - `./macos/scripts/run-e2e.sh` (or explicit scope note when macOS is intentionally excluded)
- Assertion quality checks:
  - Each step asserts visible UI outcomes (not just app launch/no crash).
  - Backend-coupled checks are tied to run list/detail, approval state transition, and output/log visibility evidence.
  - Sign-out test validates removal of user-scoped workspace/session remnants from UI/cache-facing surfaces.
- Artifact validation:
  - Confirm xcresult + test logs are discoverable via script output and/or CI upload notes in docs.

## Known Risks
- Terminal branch ambiguity if 0188 rollout status differs by environment; mitigated by explicit branch logic in trace and tests.
- Over-seeding risk can hide real regressions; mitigated by trimming seed data and preferring discovery flows.
- Flaky async UI states (run discovery/approval propagation) can cause nondeterminism; mitigated with bounded polling on explicit visible conditions.
- Possible missing accessibility identifiers could block stable selectors; mitigated by minimal targeted identifier additions coordinated with view owners.
- Local/CI environment drift (simulator, Plue health, feature flags) may impact one-command reliability; mitigated by runner preflight checks and clear failure artifact paths.
