# Implementation Plan — 0189 iOS Workspace Session Happy Path

## Scope
- Complete the iOS workspace happy path without E2E seed env dependencies: select repo/workspace context, create or resume agent session, send a user message, and observe resulting workflow runs/approvals from backend state.
- Keep terminal attach internals out of scope (ticket 0188).
- Align chat send behavior with documented implicit dispatch contract: posting a `user` message to a session dispatches the run; no explicit dispatch call.

- Locate current session selection, chat mount guards, and run list context sources.

## Validation Strategy
- Automated:
  - `xcodebuild -project SmithersGUI.xcodeproj -scheme SmithersiOS -destination 'platform=iOS Simulator,name=iPhone 16,OS=18.6' -configuration Debug CODE_SIGNING_ALLOWED=NO test`
- Behavioral checks in tests/assertions:
  - Workspace with no session shows actionable “start session” flow.
  - Existing sessions are resumed/selected deterministically.
  - Message send succeeds without `PLUE_E2E_AGENT_SESSION_ID`.
  - Runs list queries active workspace/repo context without `PLUE_E2E_REPO_OWNER` / `PLUE_E2E_REPO_NAME`.
  - “Posted message, run appears later” state transition is covered.

## Known Risks
- Session determinism ambiguity if backend session metadata lacks a clear recency/resumability signal; may require explicit tie-breaker rules.
- Eventual consistency lag between message POST success and shape/SSE run visibility can produce confusing UX if pending state/timeout copy is weak.
- Coupling across shell/chat/runs context may introduce regression in deep-link or tab-switch flows if handoff ownership is not centralized.
- Test flakiness risk in delayed run discovery scenarios; requires controlled fakes/mocks and bounded timing assertions.
