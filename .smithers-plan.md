## Known Risks
- Existing flaky UI/state timing tests may fail once assertions become strict.
- Some “BUG” tests may encode unclear expected behavior; ticket linkage may be needed before asserting.
- Accidental overlap with in-flight feature edits in the same files can cause merge friction.
- Overuse of `XCTExpectFailure` can preserve weak signal if not tightly scoped to specific assertions.

## Risk Mitigations
- Convert deterministic cases first; isolate fragile cases with minimal, explicit `XCTExpectFailure` scope.
- Use precise failure messages and ticket IDs for unresolved behavior.
- Keep edits confined to first-scope files and one inventory note.
- Re-run grep and tests after each file batch to catch regressions early.
