# 0190 iOS Onboarding Settings and Account Management — Implementation Plan

## Scope
- Expand first-run onboarding/sign-in framing to explain remote sandboxes, account requirement, and workspace lifecycle expectations.
- Add signed-out support/legal affordances (support, privacy policy, terms) on first-run/sign-in surfaces.
- Extend Settings with clearer account identity state handling (signed-in identity and explicit fallback), support/contact action, diagnostics export entrypoint, build/version visibility, privacy/terms links, sign-out, and account-deletion handoff copy.
- Refine remote access gate copy/state mapping so disabled, unauthorized, backend unavailable, and offline/unknown are distinct (avoid framing backend outage as account-level denial).
- Preserve auth-token/session mechanics owned by tickets 0184/0187.

## Files To Change
- `ios/Sources/SmithersiOS/Onboarding/OnboardingCoordinator.swift`
- `ios/Sources/SmithersiOS/Settings/SettingsView.swift`
- `Shared/Sources/SmithersAuth/SignInView.swift` (copy/link integration only)
- `ios/Sources/SmithersiOS/Diagnostics/DiagnosticsBundle.swift` (only if settings-triggered export metadata/copy needs adjustment)
- `ios/Sources/SmithersiOS/Info.plist` (only if URLs/strings should be config-driven)
- `ios/Tests/SmithersiOSTests/*` (add/extend tests for first-run, signed-in settings, signed-out settings, backend unavailable copy)

## Sequencing
1. Confirm product copy inputs and URL constants strategy
- Decide temporary defaults vs required config for support/privacy/terms/account-deletion URL.
- Align wording with ticket 0183 naming allowlist.

2. Onboarding and sign-in surface updates
- Update onboarding step content to include remote sandbox model and workspace lifecycle.
- Add signed-out legal/support links on onboarding/sign-in entry surface.
- Keep accessibility identifiers stable or add explicit IDs for new links.

3. Settings account/support/legal/account-management updates
- Ensure account section shows identity values when available and clear “not provided/unavailable” fallback when partial/missing.
- Add support/contact affordance and account deletion handoff text/link as required.
- Keep sign-out and delete-account flows functionally unchanged except copy/handoff wiring.

4. Remote access gate messaging correction
- Update signed-in gate messaging logic/copy so each state has distinct user-facing language:
  - disabled (feature off)
  - unauthorized (account not allowed)
  - backend unavailable (service outage)
  - offline/unknown (network/indeterminate)
- Ensure backend-down does not reuse unauthorized string.

5. Tests
- Add/adjust ViewInspector/unit tests for:
  - signed-out first-run links
  - signed-in settings identity + fallback
  - signed-out settings state
  - backend unavailable copy path
- Update existing feature-flag gate tests to assert corrected wording.

## Validation Strategy
- Run targeted iOS tests while iterating (onboarding/settings/feature-flag gate test classes).
- Run full ticket verification:
- `xcodebuild -project SmithersGUI.xcodeproj -scheme SmithersiOS -destination 'platform=iOS Simulator,name=iPhone 16,OS=18.6' -configuration Debug CODE_SIGNING_ALLOWED=NO test`
- Manual simulator checks:
- first run signed-out screen shows support/privacy/terms links
- signed-in settings renders identity/fallback correctly
- diagnostics export action presents share sheet or explicit error
- backend unavailable state messaging is distinct from account unauthorized

## Known Risks
- Product inputs (support/privacy/terms/account-deletion destination) may be unresolved; hardcoded placeholders can cause follow-up churn.
- Remote-access state taxonomy may span multiple models/views beyond current test coverage; partial updates could leave inconsistent copy.
- UI tests relying on exact strings/accessibility identifiers may fail if copy changes are not synchronized.
- Diagnostics export behavior depends on OS log/network capture availability; simulator/environment differences can produce flaky expectations.
