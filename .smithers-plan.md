# 0194 iOS Performance, Memory, and Rendering Pass — Implementation Plan

## Scope
- Reduce high-frequency UI publication churn on iOS terminal and chat paths without changing visible behavior.
- Keep terminal buffer bounded at 64 KiB (or replace only with an equally bounded, tested policy).
- Prevent feature-flag refreshes from republishing unchanged effective state.
- Add lightweight instrumentation/test hooks proving publication coalescing and lower update counts.
- Coordinate with ticket 0188 before any terminal mount lifecycle changes; this ticket avoids mount behavior edits unless required.

## Files To Change
- `TerminalSurface.swift`
- `ios/Sources/SmithersiOS/Terminal/TerminalIOSRenderer.swift`
- `ios/Sources/SmithersiOS/Terminal/TerminalIOSCellView.swift`
- `ios/Sources/SmithersiOS/Chat/AgentChatView.swift`
- `Shared/Sources/SmithersAuth/FeatureFlagsClient.swift`
- Tests in `ios/Tests/SmithersiOSTests/` (primary) and/or `Tests/SmithersGUITests/`

## Implementation Sequence
1. Terminal coalescing in shared model (`TerminalSurface.swift`)
- Introduce a coalescing stage between PTY `onBytes` and `@Published recentBytes` updates.
- Accumulate incoming byte chunks in a pending buffer and flush on a short cadence (e.g., main-actor timer/task sleep window) and/or size threshold.
- Preserve existing 64 KiB cap semantics on the committed buffer and ensure cap still holds under burst input.
- Add minimal counters/test hooks (guarded for tests/debug) for:
  - incoming PTY chunk count
  - published render update count
  - bytes dropped due to cap

2. iOS renderer update minimization (`TerminalIOSRenderer.swift`, `TerminalIOSCellView.swift`)
- Ensure renderer consumes coalesced model updates without reprocessing full text/grid when signature/content is unchanged.
- Add cheap equality/signature checks for snapshot/text assignment to avoid redundant `UITextView`/draw churn.
- Keep user-visible behavior intact (autoscroll, cursor, input).

3. Chat polling/backoff improvements (`AgentChatView.swift`)
- Replace fixed 2s aggressive polling loop with adaptive polling:
  - faster immediately after send/expected activity
  - backoff when repeated refreshes return no new messages
  - reset interval when new data arrives or send succeeds
- Avoid republishing message arrays when combined output is unchanged (Equatable/content-id based guard).
- Preserve optimistic send UX and existing error handling.

4. Feature flags no-op publish suppression (`FeatureFlagsClient.swift`)
- In `apply(snapshot:at:)`, only assign `snapshot` when effective state changes.
- Always update cache metadata (`cachedAt`, `lastRefreshAt`) even for no-op snapshots.
- Keep `refresh(force:)` semantics and error behavior unchanged.
- Add optional publication counter/test hook if needed for deterministic assertions.

5. Tests and evidence
- Add terminal burst test: high N incoming PTY chunks produce significantly fewer `recentBytes` publications than chunks while preserving ordered output and 64 KiB cap.
- Add feature flag refresh test: repeated unchanged remote payload updates cache timestamps but does not emit redundant `snapshot` publications.
- Add chat polling behavior test: polling interval backs off on idle and avoids fixed aggressive cadence when no new data arrives; resumes faster when activity returns.
- Run existing terminal/chat tests to confirm no UI regression.

## Validation Strategy

- Targeted assertions:

  - unchanged feature snapshots do not trigger `@Published snapshot` observer events
  - chat idle path shows increasing poll interval / fewer refreshes over fixed time window
- Closeout note:
  - capture before/after counters for terminal publish ratio, chat refresh cadence, and feature-flag publish count.

## Known Risks
- Coalescing latency tradeoff: too-large flush window can make terminal feel laggy; mitigate with small window + immediate flush on threshold/line-ending triggers if needed.
- MainActor/task lifecycle complexity: pending flush tasks must cancel cleanly on detach/deinit to avoid stale publishes.
- Equality guards can hide legitimate updates if signatures are incomplete; include conservative signatures and regression tests.
- Chat backoff tuning can delay assistant follow-ups under low traffic; reset aggressively on send/new message.
- Test flakiness from timing-based async behavior; prefer injected clock/sleeper abstractions or deterministic test hooks.
