# Implementation Plan — 0191 iOS Observability and Crash Reporting

## Scope
- Add an iOS-usable structured logging facade aligned to existing subsystem/category conventions.
- Replace highest-value iOS/shared runtime `NSLog` and direct `Logger` usage with the facade in auth/chat/runtime/PTY/feature-flag paths.
- Introduce local in-app diagnostics counters for:
  - auth failures
  - chat send success/failure
  - runtime reconnects
  - PTY attach failures
  - feature-flag refresh failures
- Add a diagnostics export bundle containing:
  - sanitized recent logs
  - counters snapshot
  - app/build/runtime metadata
- Decide beta stance for automatic remote crash/error reporting and document outcome:
  - If in scope: wire selected SDK/endpoint with privacy-safe payloads.
  - If not in scope: make manual export/reporting path explicit and reliable.
- Add/update tests for redaction and diagnostics bundle shape.
- Update tester-facing docs for diagnostics submission.

## Files To Change (expected)
- Shared logging:
  - `AppLogger.swift` (refactor) or new shared logger file colocated with existing logging utilities.
- iOS diagnostics feature:
  - `ios/Sources/SmithersiOS/Diagnostics/` (new files for counters store, bundle model/exporter, redaction helpers).
- Settings surface:
  - `ios/Sources/SmithersiOS/Settings/SettingsView.swift` (diagnostics/telemetry UI only; coordinate with ticket 0190).
- Runtime metrics emission:
  - `TerminalSurface.swift` (metric emission only; no behavior changes).
- Targeted iOS/shared runtime call sites (auth/chat/reconnect/PTY/feature-flags) where `NSLog`/direct `Logger` are replaced.
- Tests:
  - iOS/shared test targets covering log redaction and diagnostics bundle schema/content.
- Docs:
  - README or iOS release/testing docs describing how testers export and send diagnostics.

## Sequencing
1. Baseline audit
- Identify current logging primitives and high-value call sites (`NSLog`, direct `Logger`, UI-only error strings).
- Enumerate existing subsystem/category naming to keep parity with macOS patterns.

2. Logging facade enablement
- Implement/extend shared logger interface for iOS with standardized categories.
- Provide minimal adapters so existing call sites can migrate incrementally.

3. Counter instrumentation
- Implement thread-safe local counters store in `Diagnostics`.
- Wire counter increments at required events (auth/chat/reconnect/PTY/feature-flags).

4. Diagnostics export bundle
- Implement export model + serializer (e.g., JSON + optional attachment artifacts).
- Include app/build/runtime metadata and bounded recent logs.
- Add mandatory redaction pipeline before persistence/export.

5. Replace high-value call sites
- Migrate selected iOS/shared runtime logs to facade.
- Remove or reduce duplicated user-facing-only error strings where structured logs should exist.

6. Settings integration
- Add diagnostics export action and status/error UI in `SettingsView`.
- Keep edits constrained to diagnostics/telemetry area; rebase/coordinate with 0190 as needed.

7. Crash/error reporting decision + implementation/docs
- Record explicit beta decision in docs.
- If “yes”: add SDK/endpoint wiring + privacy guardrails/tests.
- If “no”: ensure manual export flow is robust and clearly documented for testers.

8. Tests and stabilization
- Add/expand tests for:
  - redaction of tokens/bearer/refresh/Authorization data
  - diagnostics bundle shape and required fields
  - counters inclusion and expected key names
- Run verification commands and resolve failures.

## Validation Strategy
- Automated verification per ticket:
  - `xcodebuild -project SmithersGUI.xcodeproj -scheme SmithersiOS -destination 'platform=iOS Simulator,name=iPhone 16,OS=18.6' -configuration Debug CODE_SIGNING_ALLOWED=NO test`
  - `swift test`
- Focused assertions:
  - Exported diagnostics never contain sensitive auth strings (`token`, `bearer`, `refresh token`, `Authorization`).
  - Export includes sanitized recent logs, counters, and app/build/runtime metadata.
  - Counter values change at expected event hooks.
- Manual QA:
  - Trigger representative failures/successes and verify counter increments + export output.
  - Validate Settings diagnostics flow (export success/failure handling and user guidance).

## Known Risks
- Parallel edits in `SettingsView.swift` with ticket 0190 may cause merge conflicts or UI regressions.
- Logging migration may miss low-visibility paths, leaving partial observability coverage.
- Over-redaction could remove useful debugging context; under-redaction is a privacy/compliance risk.
- If remote crash reporting is adopted, SDK footprint/privacy review and release gating could expand scope.
- Runtime metric hooks in shared paths (e.g., `TerminalSurface`) risk unintended side effects if not kept strictly observational.
