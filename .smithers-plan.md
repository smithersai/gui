# 0184 Shared Authenticated HTTP Client — Implementation Plan

## Scope
- Deliver a single shared authenticated HTTP abstraction in `SmithersAuth` that all surfaces can adopt for bearer-protected requests.
- Ensure behavior is refresh-aware: first `401` triggers exactly one serialized refresh path via existing `TokenManager.performWithRetry`, then request replay with refreshed bearer.
- Normalize refresh/session failures to an auth-expired signal while preserving non-auth HTTP responses (status/body/headers) for caller-owned handling.
- Keep this ticket limited to core client + tests + minimal wiring; defer broad caller migration to follow-on tickets (0186/0187/0188).

## Files To Change
- `Shared/Sources/SmithersAuth/AuthenticatedHTTPClient.swift`
  - Define/adjust public API (`send(_:)` and/or request-builder variant), response envelope type, error mapping, and docs for adoption pattern.
  - Inject dependencies (`TokenManager` protocol abstraction + `HTTPTransport`) for deterministic tests.
- `Shared/Tests/SmithersAuthTests/AuthenticatedHTTPClientTests.swift`
  - Add/maintain unit coverage for concurrent `401` collapse, successful refresh replay, refresh failure auth-expired mapping, and passthrough of `429`/`500`.
- Optional minimal wiring only if needed:
  - `Shared/Package.swift` (only if new source/test files require explicit target membership updates).

## Sequencing
1. Confirm baseline behavior and interfaces in `TokenManager.performWithRetry` and `HTTPTransport` to align retry and replay semantics.
2. Implement/adjust `AuthenticatedHTTPClient` public contract:
   - Accept prebuilt `URLRequest` and replay-safe request builder.
   - Attach bearer for each attempt.
   - Return full response envelope for all non-auth statuses.
   - Treat `401` as retry trigger only on first attempt through `performWithRetry`.
3. Map token/refresh failures (`notSignedIn`, refresh/persistence failures, whitelist denial) to `authExpired` sign-out signal.
4. Add/adjust unit tests with deterministic fake transports/token flows, including concurrent request race coverage.
5. Run test suite and fix any regressions.

## Validation Strategy
- Required commands:
```sh
cd Shared && swift test
swift test
```
- Test assertions must prove:
  - Two concurrent `401` protected requests collapse into one refresh call.
  - Successful refresh retries original request with refreshed access token.
  - Refresh failure returns auth-expired and does not loop retries.
  - Non-auth errors (`429`, `500`) are returned unchanged (status/body/headers) and do not force sign-out.
- API docs check:
  - Public caller-facing docs in the client describe intended adoption and error-handling expectations.

## Known Risks
- Replay safety: non-idempotent or streaming request bodies may not be safely replayable if callers use prebuilt requests; builder API must be documented and preferred where needed.
- Error classification drift: over-broad mapping to `authExpired` could hide transport/backend faults; keep mapping constrained to token/refresh lifecycle errors.
- Concurrency assumptions: collapse-to-one-refresh depends on `TokenManager` serialization guarantees; tests must keep explicit concurrent coverage.
- Adoption lag: until follow-on tickets migrate callers, legacy codepaths may still use direct bearer handling and inconsistent `401` behavior.
